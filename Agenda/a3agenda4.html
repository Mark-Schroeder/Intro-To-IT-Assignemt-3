Plan & progress

Aim

  • Specific aim for project and similar goals that will help achieve aim, describe these as best you can (each project should have a specific aim).

  • What are the most important parts of the project?

  • Which parts have priority over others?

    • If we have only enough time or resources for one of our goals, which one should it be?

Track progress

  • What will the project do?

  • How will we achieve a successful project?

  • How far have we got developing features or outcomes from our project?

  • Tell the story of the project.

    • How it began

    • How it progressed

    • What stage of the plan are we up to

    • Include deadends we have followed, decisions made, and changes to the project plan.

Build a roadmap with x amount of phases

  • Include roles and when they change each week/meeting

  • Define roles for particular participants, e.g. lead developer, technical designer, UI designer,

  • Roles should change week to week, depending on what needs to be done next.

  • Have you defined any specific roles for your project, If so describe and justify these.

Scope and Limits

  • Define scope and limits of the project

    • Must have vs nice to have vs out of scope

Build a proof-of-concept

  • Working software of one or two features.

  • Think of project deliverables, what outcomes can we physically show someone?

  • Include what will not be in the current project, features impossible to integrate into the app at this current time.

  • Discuss possibilities of scope creep – incorporating too many features into the project

Tools and technologies

  • Which software and tools are required by the project?

    • Are there any software licenses needed?

  • Is there any hardware needed?

  • Include any prior experience any group members have had with the tools and technologies listed.

Testing

  • How will you test your project?

  • How will you know when you will succeed?

  • Don’t leave until very end, have a quick dirty mockup

  • Build test cases for users

    • How many people will we need?

    • What background is required?

Timeframe

  • How much time to allow components?

  • Weekly detail action items for each member to track progress

    • Include action items from weeks 1-6

  • Identify risks for project

    • Software and hardware limitations

      • Difficult to learn

      • Poor documentation

      • False features

    • Knowledge transfer

  • What will we do if a group member does not respond to communications
